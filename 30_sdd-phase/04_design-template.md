# 30_sdd-phase / 04 Design テンプレート

> design.md は requirements.md の AC を満たす実装設計を記述する文書。
> 「何を作るか」ではなく「どう作るか」に集中し、後段の tasks.md の前提となる。

## 1. 文書配置

- 配置先: `specs/<feature-set>/design.md`
- フロントマター必須: `feature_ids:` 配列、`requirements_ref:` で対応 requirements.md を相対参照。
- 1 ファイル 800 行を超える前に分割 (アーキ概要 / データ / API / エージェント設計 など)。

## 2. 標準セクション (順序固定)

| # | セクション | 役割 |
|---|------------|------|
| 1 | アーキテクチャ概要 | 全体像と外部依存の俯瞰 |
| 2 | データ設計 | スキーマ、永続化方針 |
| 3 | API 設計 | 入出力契約 |
| 4 | AI エージェント設計 | LLM 呼び出し責務分割 |
| 5 | エラー処理方針 | 例外戦略、リトライ |
| 6 | 非機能設計 | 性能・可用性・セキュリティ |
| 7 | 設計判断ログ | 採用/不採用の理由 |

## 3. セクション別の書き方

### 3.1 アーキテクチャ概要

- 図 (Mermaid 等) でコンポーネント関係を示す。
- 各コンポーネントに 1〜2 行の責務説明を付ける。
- 外部サービス (DB / LLM / Object Storage) は別形状で区別する。

### 3.2 データ設計

- テーブル/コレクション/ファイル形式を列挙。
- 各エンティティに「主キー、必須フィールド、関連、インデックス、保持期間」を明記。
- マイグレーション方針 (前方互換性、バックフィル戦略) を 1 段落で記述。

### 3.3 API 設計

- HTTP API の場合: メソッド / パス / リクエスト / レスポンス / エラーコード。
- 内部関数の場合: シグネチャ / 副作用の有無 / スレッドセーフティ。
- 入力バリデーション規則を明示 (BP-027 の「証拠提示」原則と整合)。

### 3.4 AI エージェント設計

LLM を使う場合は以下を明記する。

| 項目 | 記述例 |
|------|--------|
| 役割分割 | Orchestrator / Worker の責務 (BP-005) |
| モデル割当 | Worker は安価モデル、Orchestrator は推論強モデル (BP-002, BP-018) |
| ツール権限 | エージェントごとの allowlist (BP-011) |
| プロンプト構造 | システムプロンプト / 動的コンテキストの分離 (BP-010) |
| フェイク注入点 | テスト時に置換するインタフェース (`35_tdd-phase/04_fake-injection.md` 参照) |
| 失敗モード | リトライ、フォールバック、人手エスカレーション |

### 3.5 エラー処理方針

- 例外階層を定義 (アプリ例外 / インフラ例外 / 利用者入力エラー)。
- リトライ対象とリトライ不可の境界を明示。
- 利用者向けメッセージとログ向け詳細を分離 (機密情報のログ漏洩を防ぐ)。

### 3.6 非機能設計

- requirements の非機能 AC をどう実装で満たすかを 1:1 で対応付ける。
- 性能予算 (latency budget) は呼び出しチェーンに沿って分配。

### 3.7 設計判断ログ

- 採用した方式と却下した代替案を箇条書きで残す。
- フォーマット: `- 判断: <採用案> / 却下: <代替案> / 理由: <2-3 行>`
- 後日「なぜそうなっているのか」を追体験可能にする (Architecture Decision Record の簡易版)。

## 4. requirements との対応

- design.md の各機能ブロック冒頭に「対応 AC: AC-01, AC-03, AC-05」を明記。
- AC 追加時は対応する設計セクションも更新する。

## 5. アンチパターン

- アーキテクチャ図のみで詳細が空 (図倒れ)。
- AI エージェント設計でモデル名を伏せる (再現性が損なわれる)。
- 設計判断ログがなく「なぜこの方式か」を後追いできない。
- requirements の AC と設計の対応が抜ける。

## 6. レビュー観点 (Gate 2 に連動)

- [ ] 各 AC に対応する設計箇所が特定できる
- [ ] 外部依存とフェイク注入点が明示されている
- [ ] エラー処理が「握りつぶし」になっていない
- [ ] 設計判断ログに採用理由が書かれている

## 不確実性

- アーキテクチャ図の表記法 (Mermaid / PlantUML / 画像) はプロジェクト裁量。本書は「図と責務説明をセットで残す」原則のみ規定する。
- データ設計の詳細度はチーム規模で変動するため、最低限の項目 (主キー、必須フィールド、関連、インデックス、保持期間) のみ規定する。
