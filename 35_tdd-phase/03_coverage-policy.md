# 35_tdd-phase / 03 Coverage Policy

> テストカバレッジ 80% を最低ラインとし、CI でゲーティングする。
> ただしカバレッジは「目的」ではなく「最低限の網羅証拠」として扱い、AC カバレッジ表 (`30_sdd-phase/05_tasks-template.md`) と併用する。

## 1. しきい値

| 指標 | 最低値 | 目標値 |
|------|-------|--------|
| Line coverage (全体) | 80% | 90% |
| Line coverage (新規/変更コード) | 90% | 95% |
| Branch coverage (全体) | 70% | 80% |

- 「全体」は対象モジュール全体に対する累積値。
- 「新規/変更コード」は PR 差分に限定した値 (PR レベルで強制)。

## 2. 計測対象と除外

### 2.1 計測対象

- アプリケーション本体コード (`src/` 配下)。
- 共通ライブラリ (リポジトリ内)。

### 2.2 除外可能

| 除外対象 | 理由 |
|---------|------|
| 自動生成コード | 仕様の写像であり、テストは生成元側で行う |
| マイグレーションスクリプト | 1 回限りの実行。別途手順検証で代替 |
| エントリポイント (main 関数の引数解析のみ) | I/O 配線はテスト価値が薄い |
| デバッグ用 print/log ヘルパ | 本番経路に影響しない |

### 2.3 除外の手続き

- ファイル単位またはブロック単位で除外宣言する (`# pragma: no cover` 等)。
- 除外は PR レビューで確認し、無闇に拡大しない。
- 除外宣言には理由コメントを必ず添える。

## 3. CI ゲーティング

### 3.1 PR レベル

- 新規/変更コードのカバレッジが 90% を下回る PR はマージ不可。
- 全体カバレッジが直近 main から 1 ポイント以上低下する PR は警告 (人手判断で許容可)。

### 3.2 main ブランチ

- 全体カバレッジ 80% を下回ったらビルド失敗。
- 本数値は段階的に引き上げ可能 (運用が安定したら 85%, 90% に上げる)。

### 3.3 Stop hook 連動 (BP-015)

- Claude Code のセッション終了時に Stop hook でテスト + カバレッジを実行。
- しきい値割れの状態でセッションを終わらせない。

## 4. カバレッジが嘘をつくケース

カバレッジが 100% でも以下は検出できないため、AC カバレッジ表との併用が必須。

- 行は実行されたが、アサーションが弱く挙動が検証されていない。
- 例外パスを通過したが、例外内容を assert していない。
- ハッピーパスのみで、エッジケース (空入力 / 上限値 / null) が漏れている。
- LLM のフェイクが「常に成功」を返すだけで、失敗系が検証されていない。

→ AC カバレッジ表で「全 AC が 1 つ以上のテストに紐付く」ことを別途保証する。

## 5. 計測の運用

### 5.1 計測タイミング

- ローカル: RGR の Green/Refactor 後に毎回計測。
- CI: PR ごとに計測し、PR コメントに差分を表示。
- ダッシュボード: 週次でトレンドを観測 (低下傾向の早期発見)。

### 5.2 計測対象テスト層

- Unit + Integration のみで計測する。
- E2E はカバレッジ計測対象外 (実行コスト・ノイズ過多のため)。

## 6. しきい値の設定根拠

- 80% は経験的な「テストが意味を持つ最低ライン」。これより低いと変更時のリグレッション検出力が落ちる。
- 100% を強制しない理由: 価値の薄いテスト (getter/setter 等) を量産させない。
- 新規コード 90% を厳しめにする理由: 既存負債を理由に新規コードまで甘くしないため。

## 7. アンチパターン

- カバレッジ達成のためだけのテストを増やす (アサーションが空 / 弱い)。
- 除外 pragma を乱発して数値を底上げする。
- 全体 80% を達成したと言いつつ、新規変更部のカバレッジが 30% (差分の劣化が累積)。
- カバレッジさえ通ればよいと考え、AC カバレッジ表との対応を省略する。

## 8. 段階的引き上げのロードマップ

| フェーズ | 全体 | 新規 |
|---------|------|------|
| 立ち上げ | 60% | 80% |
| 半年運用後 | 70% | 85% |
| 1 年運用後 | 80% | 90% |
| 安定期 | 85%+ | 95%+ |

レガシーコードベースを抱える場合は立ち上げ値から始め、リファクタと並行して引き上げる。

## 不確実性

- 80% という数値は業界の慣行に倣ったもので、ドメイン (例: 安全クリティカル系) によっては引き上げが必要。本書は最低ラインのみ規定する。
- カバレッジツールの実装差 (line vs statement vs branch) は言語依存のため、本書は概念レベルで記述する。
