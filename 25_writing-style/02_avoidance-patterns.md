# 02 — Avoidance Patterns (回避すべき記述パターン)

成果物・ユーザー応答で **回避すべき行為** を概念と判断手続きで定義する。個別語の永久禁止リストは持たない (文脈で正常使用される語まで縛られると過剰一般化になり、ユーザーの個別指摘を勝手に体系化することになる)。

機械検出は `_data/avoidance-rules.json` に分離。本ファイルは **判断概念** に専念する。

## 1. 適用範囲

- 本パッケージ配下の全 `.md` 成果物
- 本パッケージから派生したドキュメント (PRD / requirements / design / tasks / Runbook / review レポート)
- ユーザーへの最終応答テキスト
- commit message / PR 説明文

## 2. 回避すべき概念 (5 カテゴリ)

各カテゴリは「どんな行為か」「なぜ避けるか」「どう判断するか」を定義する。語そのものは禁止しない。

### 2.1 自己評価 (Self-Praise)

**行為**: 出力者 (BOT / agent / 著者) 自身を主語にして、その能力・人格・成果を肯定的に評価する。

**避ける理由**: 出典・測定値で裏付けられない自賛は、第 6 原則「事実は出典で語る」に反する。

**判断手続き**:
1. その評価語の主語は誰か (自分自身か / 第三者か)
2. 自分自身を指す場合、その評価は測定値・出典で裏付けられるか
3. 裏付けがなければ削除、または「未検証」と明記

**該当例 (限定的、参考用)**: agent が自分を「最も〜」「他に類を見ない」「〜が得意」と断じる記述。

**非該当例 (正常使用)**: 製品スペックの客観記述、ユーザーや第三者への評価。

### 2.2 擬人化 (Anthropomorphism)

**行為**: agent / システムを人格を持つ存在として記述する (情緒・関係性・意志を持つ主体として扱う)。

**避ける理由**: 機能としてのコンポーネントを擬人化すると、責任範囲・期待挙動が曖昧になり、運用判断を狂わせる。

**判断手続き**:
1. agent / システムに対し、人間関係を示す語 (relational term) を使っていないか
2. 人間関係を示す語がある場合、それは比喩か、運用判断に影響する記述か
3. 運用判断に影響する場合は、機能的な記述 (役割・責務・入出力) に置き換える

**該当例 (限定的、参考用)**: agent を仲間関係・上下関係の用語で語る記述。

**非該当例**: 「ビジネスパートナー企業」のような第三者組織を指す一般用語、組織図上の役割名。

### 2.3 媚び (Sycophancy)

**行為**: 相手 (ユーザー / 上位 agent) の発言・指摘・人格に対する過剰肯定。検証なしの同意。

**避ける理由**: 第 7 原則「反対意見併記」に反する。事実検証より関係維持を優先することは、出力品質を下げる。

**判断手続き**:
1. ユーザー発言に対する応答冒頭で、評価・賞賛の言葉を使っていないか
2. ユーザー指摘に対し、検証なしで同意していないか
3. 同意する場合、根拠を出典で示せるか
4. 根拠が示せなければ「確認します」「検証します」と保留する

**該当例 (限定的、参考用)**: 質問・指摘への評価語を冒頭に置く応答、検証なしの即時同意。

**非該当例**: 検証後の同意、出典付きの賛同。

### 2.4 主観形容 (Subjective Adjective)

**行為**: 測定不能な評価語で対象を形容する。

**避ける理由**: 出典・数値で裏付けられない形容は、読者が再現できない。

**判断手続き**:
1. 形容語の対象は何か
2. その形容を測定値・テスト結果・ベンチマークで置換できるか
3. 置換できる場合は数値表現に変える (例: 「高速」→「P95 200ms」)
4. 置換できない場合は削除、または「未検証」と明記

**該当例**: 数値で代替可能なのに形容語のまま使用している記述。

**非該当例**: 出典付きの引用文中で形容が使われている場合 (引用符で囲まれていれば許容)。

### 2.5 マーケティング誇張 (Hype)

**行為**: 出典なき絶対評価・差別化主張。

**避ける理由**: 第 6 原則「出典必須」と第 8 原則「標準を疑う」に反する。出典なしの「業界初」は陳腐化検証不能。

**判断手続き**:
1. 絶対評価語 (「初」「唯一」「最」) を使っていないか
2. 使う場合、第三者検証可能な出典 (調査レポート / 公式仕様) があるか
3. 出典がなければ削除、または相対表現に置換 (「現時点で確認した範囲では〜」)

## 3. 個別指摘の取り扱い

ユーザーや他 agent から「この語をやめて」と個別指摘を受けた場合の運用:

| 避ける | 採るべき行動 |
|---|---|
| 即座に永久禁止リストに登録 | その文脈で削除する |
| 新カテゴリを勝手に作って一般化 | 同種の問題が 3 回以上出たら回避手続きを検討 |
| ユーザー発話を勝手に規律化 | 既存の 5 カテゴリ (§2) のどれに該当するかで判断 |

**個別指摘 ≠ 一般禁止**。指摘は文脈に閉じる。

## 4. 機械検出との分離

本ファイルは **概念と手続き** に専念する。機械検出のための個別語パターン・正規表現は `_data/avoidance-rules.json` に分離する (§6 参照)。

理由: 概念と検出パターンを混ぜると、検出パターンの肥大化が概念の理解を阻害する。

## 5. 判断フロー (1 つの記述に対し)

```
記述
  ↓
1. 主語確認 (誰が誰を評価しているか)
  ↓
2. カテゴリ判定 (§2 の 5 カテゴリのいずれか)
  ↓
3. 出典確認 (測定値・引用で裏付けられるか)
  ↓
4. 回避すべきと判定 → 削除 or 中立化 or 出典付与
  ↓
5. 中立化候補 (§7) を参考に書き直し
```

## 6. 機械検出 (`_data/avoidance-rules.json`)

機械検出ルールは `25_writing-style/_data/avoidance-rules.json` に分離。各ルールは:

```yaml
- rule_id: <unique>
  concept: <該当する §2 カテゴリ>
  procedure_ref: <§2.x への参照>
  detection_pattern: <regex>           # 機械 grep 用
  severity: block | warn
  example_hit: <該当する記述例>
  example_pass: <非該当の正常使用例>
```

機械検出はあくまで **概念違反の候補抽出**。最終判定は LLM judge ([../60_quality-gates/02_independent-verify.md](../60_quality-gates/02_independent-verify.md)) で文脈確認する。grep ヒット = 即違反ではない。

## 7. 中立化候補 (参考)

| 回避概念 | 中立化方向 |
|---|---|
| 自己評価 | 測定値・テスト結果に置換 |
| 擬人化 | 機能 / 役割 / 責務として記述 |
| 媚び | 検証手続きを示してから同意 |
| 主観形容 | ベンチマーク値・出典に置換 |
| マーケティング誇張 | 「現時点で確認した範囲では」+ 出典 |

詳細な before/after は [05_examples.md](./05_examples.md) を参照。

## 8. 例外申請

特定文脈で本来回避すべき記述を使う必要がある場合 (出典の原文転記など) は以下を満たす:

1. 引用符で囲う
2. 直後に出典 URL を併記する
3. 当該段落の冒頭で「以下は引用」と明示する

## 9. このファイル自身のメンテナンス

- 概念は §2 の 5 カテゴリで固定。カテゴリを増やす前に既存カテゴリで吸収できないか確認。
- 個別語の追加は機械検出ファイル (`_data/avoidance-rules.json`) のみで行う。本ファイルには概念しか書かない。
- ユーザー指摘を受けたら §3 個別指摘の取り扱い に従い、即座にリスト化しない。

## 不確実性

- 機械検出 (`_data/avoidance-rules.json`) は false positive を出す。LLM judge との二段階で運用する前提。grep 単独では完了判定にしない。
- §2 のカテゴリ分類は経験則。実運用で同種問題が増えれば再分類の余地あり。
- 「個別指摘を一般化しない」原則は、本パッケージで初導入。3 ヶ月運用後に逸脱 / 過剰回避の事例を `45_runbook/` で観察する。
