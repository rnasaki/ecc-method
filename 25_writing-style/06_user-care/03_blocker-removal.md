# 03 — Blocker Removal (阻害要因の典型と即時アクション)

[02_first-response-protocol.md](./02_first-response-protocol.md) Step 4 で「阻害要因 1 つに絞る」際の参照表。典型カテゴリと即時アクション (1〜3 手) を定義する。

## 1. 阻害要因の 5 カテゴリ

| カテゴリ | 兆候 | 第一手の方向 |
|---|---|---|
| 認識ずれ | 同じ訂正 ≥ 2 / 単独訂正 / 訂正の前置き | 1 行復唱で再同期 |
| 権限不足 | 「権限がない」「承認待ち」「PR が止まる」 | 必要権限の特定と要請 |
| 失敗連鎖 | 同じエラー ≥ 3 / build / deploy / test の繰り返し失敗 | 仮説リセットして 1 軸だけ動かす |
| 情報過多 | 選択肢が多すぎて止まる / 「どれがいい?」 | 1 つに決め打ちして提示 |
| 待ち時間 | background / 外部処理の長時間化 / 進捗不明 | 状態を観測可能化、または別案で迂回 |

## 2. カテゴリ別 即時アクション

### 2.1 認識ずれ

兆候:

- 同一意図の訂正が 2 回以上来る
- 単独訂正 / 訂正の前置きの出現 ([01_emotion-detection.md](./01_emotion-detection.md) §3.1)
- ユーザー発話の主語と提案の主語がずれている

即時アクション:

1. 進行中タスク停止 ([02_first-response-protocol.md](./02_first-response-protocol.md) Step 1)
2. **1 行で要点復唱** (40〜80 字)
3. ユーザーの確認を待ってから次へ進む

NG: 「複数の解釈があり得るので A B C D のどれか…」(列挙はずれを拡大する)

### 2.2 権限不足

兆候:

- 「権限がない」「アクセスできない」のエラー
- 承認待ちで進まない
- 認証情報が古い / 失効している

即時アクション:

1. **不足している権限を 1 件特定** (例: GitHub PAT の repo scope, GCP IAM の roles/X)
2. 取得手順を 3 行以内で提示し、ユーザーに依頼
3. 取得後は事前許可リスト ([50_permissions/02_pre-authorized-actions.md](../../50_permissions/02_pre-authorized-actions.md)) に登録するか確認

NG: 全権限を一括で要求する / 関係ない scope まで含める。

### 2.3 失敗連鎖

兆候:

- 同じエラーが 3 回以上繰り返している
- build / deploy / test が同じステップで落ちる
- ログを見ずに同じ修正を繰り返している

即時アクション:

1. **仮説をリセット**: 直前の修正を撤回し、いったん原状復帰
2. **1 軸だけ動かす**: ログを 1 箇所だけ精読 (stdout / stderr / 該当エラーメッセージ)
3. **二分探索**: 動いていた最後の commit を特定し、差分を絞る

NG: 「念のため A も B も C も同時に直します」(連鎖の原因が消えない)

参照: [build-error-resolver / lang-build-resolver の委任先](../../40_delegation/01_expert-registry.md)

### 2.4 情報過多

兆候:

- 「A B C D のどれがいい?」とユーザーに選ばせている
- ユーザーから「お任せ」「適当に」が返ってくる
- 選択肢が 4 個以上ある

即時アクション:

1. **1 つに決め打ち**: 既定値を選び、理由を 1 行で示す
2. ユーザーが「中止」を返さなければ実行
3. 不可逆操作のみ事前 ASK

例:

```
NG:  「Database は Postgres / MySQL / SQLite / DuckDB のどれがいいですか?」
OK:  「Database は Postgres を採用する (理由: 既存 Runbook R-014 に整合)。やめる場合は『中止』。」
```

### 2.5 待ち時間

兆候:

- background ジョブが長時間化 (10 分超)
- 外部 API レスポンス待ち
- 進捗が見えない

即時アクション:

1. **観測可能化**: 進捗ログを 30 秒おきに出す / 状態 polling コマンドを提示
2. **別案で迂回**: 同等の結果を別経路で取れないか検討
3. **打ち切り基準**: 〇 分超過したら別案に切り替える、を事前合意

NG: 黙って待たせる / 進捗不明のまま「もう少しで終わります」を繰り返す。

## 3. カテゴリ判定の優先順位

複数カテゴリが同時に該当する場合、以下の順で 1 つに絞る。

```
1. 失敗連鎖 (放置で被害拡大)
2. 権限不足 (進めない物理制約)
3. 認識ずれ (進めても無駄になる)
4. 情報過多 (止まっているだけ)
5. 待ち時間 (進んではいる)
```

## 4. アンチパターン

| アンチパターン | 害 |
|---|---|
| 「いったん全部見直します」 | Step 4 の絞り込みに反する。リセットコスト発生 |
| 「念のため A も B も…」 | 失敗連鎖を温存する |
| 「どうしましょうか?」連発 | 情報過多をユーザーに転嫁する |
| 「もう少し待ってください」のみ | 観測可能化していない |
| 「申し訳ございません」連呼 | 阻害要因が消えない |

## 5. Runbook 化トリガ

以下のいずれかに該当したら Runbook 化を検討する ([05_post-incident-runbook.md](./05_post-incident-runbook.md)):

- 同じ阻害要因が同一案件で 2 回以上発生
- 阻害要因の解消手順が 3 ステップ以上
- 解消に 30 分以上かかった
