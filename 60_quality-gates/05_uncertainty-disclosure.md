# 05 — Uncertainty Disclosure (不確実性の開示)

不確実性を **語彙で表現** し、各成果物末尾に専用セクションを設ける義務を定める。確信と推測を読み手が区別できる状態を保つ。

---

## 1. 開示義務

以下のすべての成果物に `## 不確実性` セクションを設ける:

- ドキュメント (.md)
- spec / requirements / design / tasks
- 検証レポート
- デモ前のリリースノート

中身が「該当なし」でも空セクションを残す (隠蔽防止)。

---

## 2. 不確実性 4 区分

| 区分 | 例 | 表現 |
|---|---|---|
| 出典の鮮度 | 6 ヶ月以上経過した docs を引いた | 「retrieved_at: <date>。再検証推奨」 |
| 範囲の限定 | 1 環境のみ検証 | 「<env-A> でのみ確認。<env-B> 未検証」 |
| 推測の混入 | 公式 docs に記述なし | 「公式 docs に記述なし。実機挙動から推測」 |
| 仕組みの依存 | 動作が外部要因に依存 | 「<外部条件> が満たされる前提」 |

---

## 3. 確度語彙

主張ごとに confidence を本文で表す:

| 語彙 | 意味 |
|---|---|
| 「確認した」「検証した」 | 自分で実機 / 公式 docs で再現済み |
| 「<docs> に記述あり」 | 出典あり、実機未確認 |
| 「想定」「と思われる」 | 推測。要再確認 |
| 「現時点で確認できていません」 | 不明と明示 |
| 「未検証」 | 範囲外 |

---

## 4. NG 表現

以下は不確実性を消すため禁止:

| NG | 理由 |
|---|---|
| 「動くはずです」 | 検証なしの楽観 |
| 「たぶん大丈夫です」 | 確度不明 |
| 「一般的には〜」(出典なし) | 一般化の根拠不明 |
| 全文断定形なのに出典が薄い | 確信の偽装 |

---

## 5. `## 不確実性` セクションの書き方

```markdown
## 不確実性

- <未検証 / 範囲限定 / 推測 / 依存> の項目を箇条書き
- 各項目は「<事実>」「<再確認方法 / 期限>」を含む
```

例:

```markdown
## 不確実性

- ECC エコシステムは更新頻度が速い。本記述は 2026-06-23 時点の公式情報に基づく。四半期ごと再検証。
- aggressive プリセット内の `Bash(npm *)` は副作用が広い。隔離環境前提。
- PII 検出正規表現は文脈依存の人名検出を含まない。LLM 補助が必要。
```

---

## 6. 不確実性が積み上がったら

各成果物の `## 不確実性` 項目数が **5 件以上** になったら:

- 設計が固まりきっていないサイン
- spec 段階に戻して PRD / requirements を見直す ([55_verification/02_drift-detection.md](../55_verification/02_drift-detection.md))
- もしくは scope を縮小して確実性を上げる

---

## 7. ユーザー応答時の確度表現

ユーザーへの応答でも本章の語彙を使う:

```
「<事実>。出典 <URL> retrieved_at: <date>」
「<事実>。L1 docs 未確認。要再確認」
「現時点で確認できていません。<確認方法> で確認しますか?」
```

「自信なくはないが言い切ると怖い」事項は **確度を下げる語彙** で出す。断定も逃げも避ける。

## 9. 自動チェック

CI で以下を強制:

- 各 .md に `## 不確実性` セクションが存在
- セクションが空 (`(該当なし)` 文字列のみ含めて空) を allow するが、長文ドキュメントで空はレビュー対象
- 「動くはずです」「たぶん」等の禁止表現の grep

---

## 10. 反パターン

| 反パターン | 害 |
|---|---|
| `## 不確実性` セクション省略 | 確度が読み手から不可視 |
| 「特になし」一行で終わる長文ドキュメント | 隠蔽の疑い、レビュー対象 |
| 不確実性を別ファイルに切り離す | 該当成果物との対応が崩れる |
| 確度語彙をまぜこぜ (「動くはず」と「検証済み」を混在) | 信頼性が壊れる |
